# План: инвентарь Fun-Stats

**Дата:** 2026-05-04
**Цель документа:** инвентарь — собрать все возможные варианты fun-stats для публичного профиля и установить лимит символов, в рамках которого формулировки не обрезаются.

> Документ описательный. Никаких «что входит / что не входит / переделать» — только инвентарь. Решения о приоритете и финальных формулировках выносятся отдельно.

---

## Где живёт fun-stat в вёрстке

`app/web/templates/public_profile.html` → блок `.stats-card` под стоимостью коллекции.

```
[ stats-card ─────────────────── ]
[ СТОИМОСТЬ КОЛЛЕКЦИИ            ]
[ 204 841 ₽                      ]
[ +4 200 ₽ за месяц (delta)      ]
[ ┌──────────────────┐           ]
[ │🎧│ <текст fun-stat>│  ← ROTATE]
[ └──────────────────┘           ]
```

`fun-stat` = `[icon 26×26] [gap 10px] [text]`.

## Замеры ширины (текущая разметка, mobile-first)

| Контекст | Ширина viewport | Ширина `.stats-card` (внутри) | Доступно под текст fun-stat |
|---|---:|---:|---:|
| iPhone SE | 375 px | ~210 px | ~165 px |
| iPhone 12/13/14 | 390 px | ~225 px | ~180 px |
| iPhone Pro Max | 430 px | ~248 px | ~205 px |
| Узкий планшет (768) | 768 px | ~420 px | ~370 px |

Шрифт `Inter Tight 13px`, средняя ширина символа ~7 px (русский, без acccents). Line-height 1.35.

| Раскладка | Доступно | Чистых символов в строку |
|---|---:|---:|
| 1 строка, iPhone SE (165 px) | 165 / 7 | **~22** |
| 1 строка, iPhone 12+ (180 px) | 180 / 7 | **~25** |
| 2 строки (с переносом) | 2 × N | **~44–50** |

Выбор: **жёсткий лимит 50 символов на fun-stat**, целимся в 25–35 (одна-две строки на любом мобильном).

## Лимиты символов по позициям внутри строки

`<icon> <число><спейс><метка>`
- Число: 1–4 цифры (`<b>17</b>`, `<b>2025</b>`).
- Метка: остальное.

| Длина числа | Бюджет на метку (50 − len(num) − 1) |
|---:|---:|
| 1–2 | 47–48 |
| 3 | 46 |
| 4 | 45 |

То есть **метка влезет в 45 символов** при любом числе. Целевая длина метки — **30 символов**.

| Длина (символов) | Кол-во вариантов | Замечание |

## Гарантия отображения

В текущей вёрстке (`public_profile.html`):
```css
.fun-stat .text {
    font-size: 13px; line-height: 1.35;
    word-wrap: break-word; overflow-wrap: break-word;
}
.fun-stats-wrap { min-height: 44px; overflow: hidden; }
```
JS поджимает `min-height` под самый длинный факт после рендера.

При лимите ≤ 50 символов гарантируется отображение в 1–2 строки на всех мобильных шириной ≥ 360 px.

## Источники данных (прямой mapping на БД)

Все агрегации строятся поверх distinct-`record_id` подзапроса:

```sql
WITH user_records AS (
    SELECT DISTINCT ci.record_id AS rid
    FROM collection_items ci
    JOIN collections c ON c.id = ci.collection_id
    WHERE c.user_id = :user_id
)
SELECT … FROM user_records ur JOIN records r ON r.id = ur.rid
```

Это уже реализовано в `app/web/routes.py` (функции fun_stats блока).

Дополнительные источники, которых в текущем расчёте ещё нет:
- D6/D7 — `CollectionItem.estimated_price_rub` (нужен JOIN, не distinct)
- D8 — `Record.discogs_want` (есть в схеме, не используется в fun_stats)
- E2/E3/E4 — `wishlist_items`, `follows`, `gift_bookings` (есть в БД)
- E5/E6 — `CollectionItem.added_at`
- F4 — `SUM(Record.discogs_want)`

## Фильтры показа

Каждая стата выводится только если значение «достаточно интересное»:

| Стата | Минимум для показа |
|---|---|
| Любой counter | `≥ 1` (для жанров `≥ 2`, для лейблов `≥ 3`) |
| Декада | `≥ 2` пластинки в одной декаде |
| Стран | `≥ 2` |
| Top artist/label | `≥ 2` пластинки этого артиста/лейбла |
| Самая старая/свежая | всегда, если есть `year` |
| F-курьёзы | свои пороги (см. таблицу) |

## Что значит «работает автоматически для каждого юзера»

Уже работает: расчёт идёт в `app/web/routes.py:public_profile_page` для любого `@username`, без какой-либо настройки. Каждый юзер, шарящий ссылку, получает свои fun-stats на основе своей коллекции.
